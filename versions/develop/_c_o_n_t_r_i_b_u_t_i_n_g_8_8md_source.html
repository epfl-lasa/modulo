5 ### Exceptions
6 
7 This section describes how a component should handle exceptions and errors.
8 
9 As a general rule, a component interface methods do not throw an exception or raise an error unless there is no other
10 meaningful option. More precisely, only non-void public/protected methods throw, i.e. all setters and `add_xxx` methods
11 do not throw but catch all exceptions and log an error.
12 
13 If an exception is thrown, it is either a `ComponentException` (in C++) or a `ComponentError` (in Python) or any
14 derived exception, such that all exceptions thrown by a component can be caught with those base exceptions (for example
15 in the periodic `step` function).
16 
17 Currently, the following methods throw:
18 - `get_parameter`
19 - `get_parameter_value`
20 - `lookup_transform`
21 
22 ### Logging
23 
24 Similar to the exceptions, the logging of debug, info, and error messages should follow some principles:
25 
26 - Methods that catch an exception and are not allow to rethrow, log an error with the exception message
27 - `add_xxx` methods use non-throttled logging
28 - Setters and getters as well as all other methods that are expected to be called at a high frequency use throttled
29  logging
